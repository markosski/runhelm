# Event-Backed Workflow State

RunHelm is considering an event-backed snapshot model for workflow instance state changes. The proposal is tracked in `openspec/changes/add-event-backed-workflow-state/`.

The intended middle ground is:

- workflow state changes are represented as ordered domain events
- user and orchestrator operations are represented as domain commands that validate the current snapshot and produce one or more durable events
- core reducer logic applies events to produce the next `WorkflowInstance` snapshot
- a core workflow state manager coordinates command handling, then asks storage to commit timestamped event records with the updated snapshot
- state manager callers commit commands by workflow instance ID, which loads the current snapshot, or against an already-held `WorkflowInstance`, avoiding redundant snapshot reads
- workflow instance snapshots carry a monotonically increasing `version`, and every commit includes the expected base version
- runtime workflow, task, and verifier state transitions commit through the workflow state manager rather than direct snapshot saves
- storage adapters persist event records and snapshots, and may maintain lightweight summary rows from committed snapshots, but do not interpret event meaning
- current workflow reads continue to use snapshots rather than replaying events
- workflow instance lists use one filtered summary query instead of returning full workflow instances
- workflow definitions can be overwritten only until the first workflow instance is created for that definition ID

This is not full event sourcing. Full replay-based reads, snapshot checkpoints, event version migrations, and durable database transaction details remain future work until RunHelm needs durable storage semantics.

The main boundary is that command handling and event processing belong in core/domain code. Commands represent requested operations, such as pausing a workflow, submitting human input, retrying a task, force retrying on another host, applying startup recovery, starting or completing a workflow run, materializing task attempts, starting a task attempt, and recording task execution results. Command payloads should contain caller intent and externally discovered data only; fields that can be derived from the current `WorkflowInstance` are derived by the command processor before emitting events. Durable events represent past-tense facts. A command validates the current snapshot and produces the explicit events needed to describe the resulting state change; the reducer applies those facts to build the next snapshot. Storage should provide persistence mechanics only, so memory, SQLite, or Postgres adapters can remain swappable without duplicating workflow transition rules. The core workflow state manager assigns `created_time` when wrapping events in `WorkflowEventRecord`; storage persists those records without interpreting the payload. The state manager also records the loaded snapshot version as the expected commit version and advances the updated snapshot version by the number of committed events. Storage rejects the commit with a typed workflow version conflict when the currently stored snapshot version no longer matches the expected version. Storage port methods return storage-scoped errors so callers can distinguish expected concurrency conflicts from backend failures. Storage adapters may build and persist `WorkflowInfo` summary rows from the committed `WorkflowInstance` snapshot without pushing optimization-only fields back into the domain snapshot. Summary lifecycle timestamps are derived from persisted workflow event record timestamps: creation from the first event batch, modification from the latest event batch, and completion from the first terminal reduced snapshot. The storage port no longer exposes a direct runtime snapshot-save operation for workflow instances.

The intended storage read shape is also narrower than the current snapshot API. `get_workflow_instance` remains the only storage operation that returns full workflow state. List calls should return lightweight summaries with identity, lifecycle timestamps when available, current state, and task completion counts. `list_workflow_info` accepts zero or more summary filters plus a required page request; an empty filter list matches all summaries, status filters match any listed status, workflow definition ID filters match one definition, and multiple filters are combined with AND semantics. Results are sorted by most recent `modified_at_epoch_ms` first, with workflow instance ID as a deterministic tie-breaker, and callers page with an opaque cursor derived from that ordering. Active workflow discovery uses the same filter and pagination mechanism as list views, requesting pending and running statuses.

Workflow definition registration remains mutable only before execution history exists. Once any workflow instance summary exists for a workflow definition ID, registering another definition with that normalized ID is rejected so historical instances cannot be reinterpreted against a newer graph. Callers should create a new workflow definition ID for versioned workflow changes after instances exist.

The public API exposes workflow events separately from the current-state workflow read:

- `GET /workflows/{id}` returns the current workflow status snapshot view.
- `GET /workflows` returns a bounded page of `WorkflowInfo` list entries with workflow identity, lifecycle timestamps when available, current state, task counts, and a `next_cursor` when more results are available. It accepts `status` to filter by one workflow status, using any status value returned by the API such as `Pending`, `Running`, `Paused`, `InputNeeded`, `Completed`, or `Failed`.
- `GET /workflows/{id}/events` returns timestamped `WorkflowEventRecord` entries for audit and debugging.
- `POST /workflows/{id}/pause` commits a pause command. For a `Pending` or `Running` workflow, the command records a `WorkflowStatusChanged` event to move it to `Paused` and the orchestrator removes that workflow from the pending execution queue. Repeating the request for an already paused workflow is accepted as a no-op. Terminal workflows and workflows waiting for human input reject pause with conflict behavior.
- `POST /workflows/{id}/resume` commits a resume command that records a `WorkflowStatusChanged` event to move a paused workflow back to `Pending`, then queues it for execution.
- `POST /workflows/pause` pauses every `Pending` or `Running` workflow visible in workflow summaries and removes each from the pending execution queue. The response includes the affected workflow instance IDs.
- `POST /workflows/resume` resumes every `Paused` workflow visible in workflow summaries and queues each resumed workflow. The response includes the affected workflow instance IDs.
- `GET /workflows/{id}/tasks` and `GET /workflows/{id}/tasks/{task_id}` report task attempts waiting for human input with `status: "input_needed"` and `input_request`; they are not reported as `running`.
- `POST /workflows/{id}/tasks/{task_id}/human-input` with `{ "input": ... }` commits a submit-human-input command for a workflow and task attempt that are currently waiting in `InputNeeded`. The command materializes the continuation attempt for the same logical task with `TaskMaterialized` carrying the submitted input, and records `WorkflowStatusChanged` to move the workflow back to `Pending` so it can be queued to continue.
- `POST /workflows/{id}/tasks/{task_id}/retry` commits a retry command for a failed workflow and failed task attempt. The command records explicit task reset facts, moves the workflow back to `Pending`, and preserves the existing workflow host pin.
- `POST /workflows/{id}/tasks/{task_id}/retry?force=true` commits a force-retry command. If the existing pinned host is currently eligible, retry keeps that host. If the existing host is unavailable and another host is eligible, retry records a `WorkflowPinnedHostChanged` event and explicit task reset facts, and marks that local context may be lost. If no retry host is eligible, the request is rejected with capacity unavailable behavior and no event is committed.

Retry is an orchestrator-owned application operation: the API handler parses the request and selects normal versus force retry, then `Orchestrator` validates retry state through workflow domain services, commits the retry command, and enqueues the workflow instance. Force retry receives the current worker-pool liveness state at the operation boundary so host selection stays with the retry operation without making worker pool long-lived orchestrator state.

Pause and resume are orchestrator-owned application operations for the same queue reason: workflow domain services validate and commit status events, while the orchestrator removes paused workflows from the pending queue and enqueues resumed workflows. Pause prevents queued or future workflow passes from dispatching more work. It does not cancel an executor call that is already in flight; that call may still finish its current attempt. If that in-flight task is not the final task, RunHelm records the task result and leaves the workflow `Paused` without dispatching downstream work. If that in-flight task completes the workflow, RunHelm records the result and moves the workflow to `Completed`. In-flight failure and human-input results may still move the workflow to `Failed` or `InputNeeded` because those states are stronger terminal or blocking outcomes than a generic pause.

Short term, `InputNeeded` is workflow-blocking: when any task asks for human input, the current engine pass stops after recording that task state. Independent branches that could otherwise run remain pending until submitted human input queues a fresh pass. The workflow queue prevents overlapping engine passes for the same workflow instance by tracking pending and active workflow instance IDs; if a workflow is enqueued while an engine pass is active, the queued ID waits for a later pass. Those scheduler guards reduce duplicate engine passes, while workflow version checks provide the storage-level protection against stale engine or API snapshots overwriting newer committed workflow state.
