Verifier-controlled bounded reruns

A workflow can revise a focused slice without restarting everything.

Bounded workflow loops let an Agent verifier reject a generation, provide feedback, and ask RunHelm to rerun a configured upstream slice. The workflow stays flat, bounded, persisted, and observable.

Acyclic Normal data bindings remain a DAG.
Bounded Each verifier has a max iteration budget.
Auditable Rejected generations stay visible.

The Definition Model

A bounded loop is not a new nested workflow language. It is a control.verifier block on a verifier task. The block defines a bounded control edge back to an upstream ancestor. 

control:
  verifier:
    max_iterations: 3
    on_exhausted_continue: false
    rerun_from_task_id: b

# The verifier task returns JSON:
# { "decision": "complete" }
# or
# { "decision": "continue", "feedback": "Improve evidence and formatting." }
Validation

Verifier task owns the decision

The verifier task returns a JSON decision. RunHelm injects that decision schema when control.verifier is declared.

Bounds

Iteration budget is explicit

max_iterations must be positive. Exhaustion either fails the workflow or finalizes the latest valid generation.

Safety

No custom output schema

A verifier task must not declare its own output_schema. RunHelm supplies the required decision schema.

Lifecycle Of A Bounded Rerun

Runtime state moves from logical task IDs to materialized attempts for the rerun slice. Rejected attempts complete normally at the task lifecycle layer, while verifier metadata records why the generation was not selected.

1

Run normal upstream tasks

Tasks outside the rerun slice, such as A, run once and keep their selected output.

2

Materialize generation 1

When the rerun start becomes eligible, RunHelm creates B[1], C[1], and D[1] instead of overwriting B, C, or D.

3

Parse the verifier decision

The verifier task output must satisfy the injected decision schema before RunHelm applies complete or continue.

4

Continue creates the next generation

A valid continue decision requires non-empty feedback. RunHelm persists that feedback and creates B[2], C[2], and D[2] if budget remains.

5

Complete selects a generation

A complete decision selects the current generation. Downstream tasks after the verifier consume only the selected output.

Binding Rules

The core implementation decision is generation-scoped resolution. The same logical binding can resolve to a different materialized attempt depending on where the consumer runs.

Consumer Source Binding Resolved Source Why It Matters
C[2] B -> C B[2] Tasks inside a rerun slice consume same-generation upstream outputs.
B[2] A -> B A Inputs from outside the slice stay stable across generations.
E D -> E D[2] after selection Downstream work waits for an accepted or exhaustion-finalized verifier generation.
Rejected D[1] D -> E Does not satisfy readiness Rejected generations remain observable but do not leak into downstream execution.

Use Case: Market Report With Format Verification

A stock-report workflow is a compact example because correctness is not just "did an Agent return JSON." The result needs fresh data, currency formatting, dates, and a useful summary.

Workflow Shape

  • fetchprices: gets recent AAPL and MSFT market data.
  • draftsummary: writes a concise market summary.
  • verify: checks date format, dollar formatting, and source clarity.
  • If verification fails, rerun from fetchprices with concrete feedback.

Why bounded loops fit this use case

The verifier can reject a weak report without restarting unrelated setup work. Feedback is injected into the next rerun generation as loop context. The worker receives the most recent feedback, ordered feedback history, and the previous same-task output, so the Agent can correct the current issue while preserving earlier corrections.

Implementation Decisions To Notice

The feature is high-level for workflow authors, but a few implementation choices make it reliable enough for production orchestration.

Materialized attempts are stable IDs

A rerun slice persists as b[1], c[1], d[1], then b[2], c[2], d[2]. The instance also keeps the original task definition ID and generation index.

Verifier state is separate from task lifecycle

A rejected generation can have task lifecycle Completed because it produced schema-valid output. Selection, rejection, exhaustion, and feedback live in verifier metadata.

Loop context is execution metadata

Feedback history is carried as execution metadata on the normal executor and worker enqueue path, not as a user-declared input slot. The workflow input schema remains about business data, not orchestration state.

Ordinary cycles are still rejected

Data bindings remain acyclic. The verifier backedge is explicit control flow and only valid from an Agent verifier to one of its upstream ancestors.

Exhaustion policy is explicit

on_exhausted_continue: false fails the workflow when budget is exhausted. true finalizes the latest schema-valid generation, and records why.

Rejected generations are observable

RunHelm does not erase failed review history or roll back side effects. Rejected attempts remain part of the persisted audit trail.

Operational guidance

Put side-effect-heavy tasks outside rerun slices when possible. If a rerun slice must perform side effects, make those effects idempotent or safe to repeat because rejected generations are retained, not rolled back.
