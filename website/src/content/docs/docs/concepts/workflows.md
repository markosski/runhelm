title: Workflows
description: Understand workflow definitions, task kinds, data bindings, schemas, and run state.
---

A RunHelm workflow is a structured definition of tasks and the data flow between them. A workflow definition becomes a workflow instance when a run starts.

## Workflow definitions

A workflow contains:

- tasks
- data bindings between task outputs and downstream task inputs
- optional per-task timeouts
- optional per-task input and output schemas
- task kinds such as `Agent`, `ApiCall`, and `Function`

Workflow definitions are JSON or YAML shaped so they can be generated by tools and eventually edited through UI flows.

## Task execution model

At runtime, the orchestrator repeatedly finds tasks whose dependencies are satisfied and dispatches them through an executor backend. Workers claim queued task payloads, execute the work, and post results back to the orchestrator.

The high-level flow is:

1. Register a workflow definition.
2. Create a workflow instance for a run.
3. Identify runnable tasks.
4. Queue runnable tasks behind an executor backend.
5. Let workers claim and execute tasks.
6. Validate task outputs when schemas are declared.
7. Propagate outputs through data bindings.
8. Persist progress so runs can be inspected and resumed.

## Run and task states

Task state describes where a task is in the run:

- `Pending`
- `Running`
- `InputNeeded`
- `Completed`
- `Failed`

The workflow run state is promoted as tasks move through these states. This keeps orchestration logic deterministic while allowing execution runtimes to evolve independently.

## Bounded loops

RunHelm supports bounded verifier-style loops for workflows such as implementation and review. A verifier task can ask the workflow to continue with feedback, causing a previous step to run again up to a configured limit.
