Agentic workflow orchestration

RunHelm separates workflow control from task execution.

The orchestrator owns definitions, scheduling, state, and HTTP visibility. Workers execute typed tasks through a narrow protocol, so agent calls, API calls, and custom functions can evolve independently.

Component Roles

RunHelm uses ports and adapters so side effects stay replaceable while workflow decisions stay concentrated in the core engine.

orchestrator/

Rust control plane. It stores workflow definitions, creates workflow instances, queues runs, exposes HTTP endpoints, and drives task state transitions through the workflow engine.

worker/

Node.js task runtime. It connects to the orchestrator socket, registers capabilities, receives task dispatch messages, selects an executor, validates outputs with AJV, and sends results back.

frontend/

React operator console prototype. It is intended to show workflow visibility, run monitoring, and operational status as the API surface matures.

Execution Flow

A run advances only when dependencies are satisfied and outputs match the declared schema. That keeps the orchestration loop predictable.

Register definition

Submit a workflow with tasks, JSON schemas, and data bindings.

Create instance

The orchestrator creates a pending instance with one task instance per task.

Find ready tasks

The engine checks bindings and runs only tasks whose upstream inputs are complete.

Dispatch work

The executor port sends task payloads to a fake, Docker, or IPC-backed worker executor.

Validate output

Output schemas are enforced before data is stored or sent downstream.

Report status

Runs become completed, failed, or input-needed and remain queryable through HTTP.

Contracts

The important boundary is the task contract: inputs, output schema, required credentials, and the selected task kind.

Agent

LLM

Uses a provider URL, model id, prompt, optional tools, and optional ask behavior. Credentials are looked up through the credentials port.

ApiCall

HTTP

Encodes network calls as workflow nodes so external API work has the same state and validation treatment as other tasks.

Function

Code

Runs task-provided JavaScript with declared dependencies and returns structured output to be schema-checked.

Operational Model

Docker Compose runs the API and multiple workers with a shared Unix socket volume. Credentials are mounted read-only into the worker.

State visibility

  • Workflow status: Pending, Running, InputNeeded, Completed, Failed
  • Task status is reported separately for each task.
  • Task outputs can be fetched after completion.

Isolation

  • HTTP traffic enters through the orchestrator only.
  • Workers communicate over the RunHelm socket.
  • Compose services run with read-only filesystems and no-new-privileges.

Credentials

  • The worker reads ~/.runhelm/file_credentials.json.
  • Credential names are declared per task.
  • Adapters keep secret lookup behind a testable interface.

How To Use It

Use Docker Compose for the integrated path, or run the orchestrator and worker separately while developing.

Local workflow

  1. Create credentials at ~/.runhelm/file_credentials.json if your tasks require provider keys.
  2. Start the system with docker compose up --build.
  3. Register a workflow definition with POST /workflow-def.
  4. Trigger a run with POST /workflow-def/{def_id}.
  5. Inspect the run with GET /workflows/{id}.
  6. Fetch a task result with GET /workflows/{workflow_instance_id}/tasks/{task_id}.
Current shape: RunHelm is early-stage. Some adapters and the frontend are scaffolds, but the core contracts and worker IPC path are already present in the codebase. 
# Start integrated services
docker compose up --build

# Health check
curl -sS http://localhost:3000/health

# Register the sample workflow
yq -o=json worker/example_workflow.yaml \
  | curl -sS -X POST http://localhost:3000/workflow-def \
      -H 'content-type: application/json' \
      --data-binary @-

# Trigger a workflow run
curl -sS -X POST http://localhost:3000/workflow-def/simple-function-workflow \
  -H 'content-type: application/json' \
  -d '{}'

# Check run status
curl -sS http://localhost:3000/workflows/<workflow-instance-id>
