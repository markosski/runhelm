Workflow Composition

Workflows are graphs of typed tasks. The engine moves data only through declared contracts, then checkpoints each accepted result so runs can be inspected and resumed from durable state.

API task

Collect external data

Calls an HTTP endpoint with declared request inputs and a required response shape.

output: customer_profile

schema binding
Function task

Transform with JavaScript

Runs custom JavaScript and can use declared npm packages for local computation.

input: customer_profile

Agent task

Reason over context

Calls an agent with structured inputs, declared available tools, and a required structured result.

input: decision_context

tools: web search, fetch, time

Definition checkpoint Task graph, schemas, bindings, credentials, and expected side effects are stored.
Dispatch checkpoint Ready tasks are recorded before work leaves the engine.
Result checkpoint Outputs are persisted only after they satisfy the task contract.
Run checkpoint Workflow status advances from durable task state, not transient worker memory.

Component Roles

RunHelm keeps workflow decisions concentrated in the engine while side-effecting work stays behind replaceable adapters.

orchestrator/

Control plane for workflow definitions, workflow instances, the pending queue, HTTP visibility, and task state transitions.

worker/

Task runtime. It receives dispatch messages, selects the right executor, validates outputs, and sends contract-shaped results back.

Execution Flow

A run advances only when dependencies are satisfied and outputs match the declared schema. That keeps the orchestration loop predictable.

Register definition

Submit a workflow with tasks, JSON schemas, and data bindings.

Create instance

The orchestrator creates a pending instance with one task instance per task.

Find ready tasks

The engine checks bindings and runs only tasks whose upstream inputs are complete.

Dispatch work

The executor port sends task payloads to a fake, Docker, or IPC-backed worker executor.

Validate output

Output schemas are enforced before data is stored or sent downstream.

Report status

Runs become completed, failed, or input-needed and remain queryable through workflow collection and detail endpoints.

Operational Model

Docker Compose runs the API and multiple workers with a shared Unix socket volume. Credentials are mounted read-only into the worker.

State visibility

  • Workflow status: Pending, Running, InputNeeded, Completed, Failed
  • Workflow instances can be listed with GET /workflows and filtered with ?status=running or other lifecycle states.
  • Task status is reported separately for each task.
  • Task outputs can be fetched after completion.

Queue control

  • GET /queue returns pending workflow instance IDs.
  • DELETE /queue/{id} removes one pending item.
  • DELETE /queue purges the pending queue without cancelling running workflows.

Isolation

  • HTTP traffic enters through the orchestrator only.
  • Workers communicate over the RunHelm socket.
  • Compose services run with read-only filesystems and no-new-privileges.

Credentials

  • The worker reads ~/.runhelm/file_credentials.json.
  • Credential names are declared per task.
  • Adapters keep secret lookup behind a testable interface.

How To Use It

Use Docker Compose for the integrated path, or run the orchestrator and worker separately while developing.

Local workflow

  1. Create credentials at ~/.runhelm/file_credentials.json if your tasks require provider keys.
  2. Start the system with docker compose up --build.
  3. Register a workflow definition with POST /workflow-def.
  4. Trigger a run with POST /workflow-def/{def_id}.
  5. Inspect the pending queue with GET /queue.
  6. List runs with GET /workflows or GET /workflows?status=completed.
  7. Inspect the run with GET /workflows/{id}.
  8. Fetch a task result with GET /workflows/{workflow_instance_id}/tasks/{task_id}.
Current shape: RunHelm is early-stage. Some adapters and the core task contracts and checkpointed execution path are already present in the codebase. 
# Start integrated services
docker compose up --build

# Health check
curl -sS http://localhost:3456/health

# Register the sample workflow
yq . worker/example_workflow.yaml \
  | curl -sS -X POST http://localhost:3456/workflow-def \
      -H 'content-type: application/json' \
      --data-binary @-

# Trigger a workflow run
curl -sS -X POST http://localhost:3456/workflow-def/simple-function-workflow \
  -H 'content-type: application/json' \
  -d '{}'

# Check run status
curl -sS http://localhost:3456/workflows/<workflow-instance-id>

# List workflow runs
curl -sS 'http://localhost:3456/workflows?status=running'

# Inspect and manage pending queue
curl -sS http://localhost:3456/queue
curl -sS -X DELETE http://localhost:3456/queue/<workflow-instance-id>
curl -sS -X DELETE http://localhost:3456/queue

Orchestrator API

The orchestrator exposes a small HTTP API on localhost:3456 by default. All write endpoints that accept data expect JSON request bodies.

GET

/health

Checks whether the orchestrator process is accepting HTTP traffic.

Response
200 OK with plain text OK.
POST

/workflow-def

Registers or replaces a workflow definition made of task definitions and data bindings.

Body
WorkflowDef: id, tasks, and data_bindings.
Response
{ "status": "created", "id": "..." }
POST

/workflow-def/{def_id}

Creates a workflow instance from an existing definition and puts it on the execution queue.

Body
Any JSON object. The current handler accepts the payload but does not consume fields from it.
Response
{ "status": "queued", "id": "..." }; 404 when the definition is unknown.
POST

/workflow-def/{def_id}/tasks/{task_id}

Runs one task from a registered workflow definition without creating a workflow instance.

Body
{ "inputs": [...] }, where each array item is passed to the task as an input value.
Response
success, input_needed, or failure; 404 when the definition or task is unknown.
GET

/queue

Lists pending workflow instance IDs waiting for scheduler pickup.

Response
{ "pending": ["..."] }
DELETE

/queue

Removes every pending workflow instance from the queue. Running workflows are not cancelled.

Response
{ "status": "purged", "purged": ["..."] }
DELETE

/queue/{id}

Removes one pending workflow instance from the queue.

Response
204 No Content; 404 when the queue item is missing.
GET

/workflows

Lists workflow instances, optionally filtered by lifecycle status.

Query
status accepts pending, running, paused, input_needed, completed, or failed.
Response
{ "workflows": [{ "id": "...", "workflow_def_id": "...", "status": "..." }] }
GET

/workflows/{id}

Returns a status report for one workflow instance, including task state and output availability.

Response
instance_id, workflow_def_id, status, and tasks; 404 when the instance is unknown.
GET

/workflows/{workflow_instance_id}/tasks/{task_id}

Fetches the current result state for one task in a workflow instance.

Response
success with output, failure with error_message, pending, or running.
